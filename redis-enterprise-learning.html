סקירה כללית — Redis Enterprise

Redis Enterprise (Redis Software) הוא פתרון מסדי נתונים מבוזר שמרחיב את Redis הפתוח עם HA, Persistence, Sharding אוטומטי, וסנכרון בין-קלאסטרי. בנוי על ארכיטקטורה סימטרית של nodes שווים, כש-כל node מריץ אותם רכיבים: Cluster Manager, Proxy, ו-Redis instances (shards).

ארכיטקטורה רשמית — 4 רכיבים מרכזיים

לפי תיעוד הטרמינולוגיה הרשמי של Redis Enterprise:

  • Node — מכונה פיזית/וירטואלית/קונטיינר עליה הותקן Redis Enterprise. כל node הוא container ל-Redis instances ("shards").
  • Zero-latency Proxy — מקור: "Each node includes one zero-latency, multi-threaded proxy (written in low-level C)". Proxy אחד לכל node, מקבל את הפעולות ומעביר אותן ל-shards. בנוי בארכיטקטורת cut-through.
  • Cluster Manager — מקור: "The cluster manager oversees all node management-related tasks, and the cluster manager in the master node looks after all the cluster related tasks". מנותק מ-Redis לחלוטין כדי לא להשפיע על ביצועים. אחראי על הקצאת shards, migration, monitoring, ו-failover events.
  • Shard — Redis instance בודד. כל database מורכב מ-shard אחד או יותר; כל shard יכול להיות master או replica.

מודל הריצה

על כל נוד מריץ supervisord בתור פרוסס שורש (PID 1 בתוך הקונטיינר/דמון). הוא מנהל את כל הפרוססים האחרים — מתחיל אותם, מנטר אותם, ומפעיל מחדש אם הם נופלים. cnm_exec (Cluster Node Manager) הוא הסוכן שמדבר עם הקלאסטר ומקבל פקודות לבצע שינויים. CCS (Cluster Configuration Store) הוא ה-store המבוזר ששומר את כל מצב הקלאסטר.

DMC Proxy רץ על כל נוד — כל קליינט שמתחבר לנוד מסוים מקבל שירות מ-proxy זה, שמנתב את הפעולה לפי ה-key אל ה-shard הנכון, ואז ל-master המתאים (אפילו אם הוא בנוד אחר).

שתי טופולוגיות סנכרון בין-קלאסטריות

Active-Active (CRDB)

כל הקלאסטרים writable. שינויים מתפזרים אסינכרונית דרך ה-CRDB Syncer. הקונפליקטים נפתרים אוטומטית באמצעות CRDTs (Conflict-Free Replicated Data Types).

מתאים ל: אפליקציות גלובליות, Multi-region writes, Latency נמוך לכל המשתמשים.

Active-Passive (Replica Of)

קלאסטר אחד primary (writable), שני secondary (read-only). הסנכרון אסינכרוני כברירת מחדל (replica-of). מתאים ל-DR (Disaster Recovery).

בתוך הקלאסטר עצמו, הסנכרון בין master shard ל-replica shard הוא אסינכרוני כברירת מחדל אך ניתן להגדיר WAIT/Sync replication per command.

המלצה: קרא את הטאבים בסדר — סקירה → ארכיטקטורה → שירותים → פורטים → Process Tree → AA → AP → Syncer → HA → Q&A.

ארכיטקטורת קלאסטר — 5 Nodes

להלן קלאסטר Redis Enterprise בעל 5 נודים. כל נוד מריץ סט מלא של שירותים: supervisord כשורש, DMC Proxy, CCS, cnm_exec, stats_archiver, pdns/mdns, envoy, sentinel, watchdog, alert_mgr. בנוסף, על כל נוד יושבים shards (Redis processes) — חלקם master, חלקם replica.

חשוב: ב-Redis Enterprise כל הנודים שווים בתפקיד — אין "master node" או "worker node". כל נוד מריץ את אותו set שירותים, ולכל נוד יכולים להיות master shards וגם replica shards. ההיררכיה הקיימת היא ברמת ה-shard (master shard / replica shard) ולא ברמת ה-node. ה-CCS Leader הוא תפקיד דינמי שנבחר בריצה. נוד יכול להיות מוגדר כ-quorum-only (לא נושא shards) — אך זו בחירת deployment ולא היררכיה.

זרימת בקשה מהקליינט אל ה-Shard

  1. קליינט מתחבר ל-FQDN של ה-DB (לדוגמה redis-12000.cluster-a.local) → DNS מחזיר IP של אחד הנודים.
  2. הקליינט שולח SET key value ל-port של ה-DB (12000).
  3. DMC Proxy על אותו node מקבל את הבקשה (multiplexing).
  4. ה-Proxy מחשב CRC16(key) % 16384 → מקבל את ה-slot.
  5. ה-Proxy מתייעץ עם CCS על המיפוי slot → shard → node.
  6. הבקשה נשלחת ל-master shard (יכול להיות בנוד אחר — תעבורה תוך-קלאסטרית!).
  7. ה-master מבצע, ושולח אסינכרונית ל-replica.
  8. תשובה חוזרת בחזרה דרך ה-Proxy לקליינט.

HA Architecture — איך מתממש High Availability

לפי Terminology — docs.redis.io

1. Master-Replica Shard

ציטוט: "Every database master shard has a replica shard, so that if the master shard fails the cluster can automatically fail over to the replica with minimal impact. Master and replica shards are always placed on separate nodes to ensure high availability."

הסנכרון בין master ל-replica הוא אסינכרוני (ברירת מחדל) באמצעות פרוטוקול replication של Redis.

2. Cluster Manager — Failover

ציטוט: "This enables RS to react in a much faster and accurate manner to failure events, so that, for example, a node failure event triggers mass failover operations of all the master endpoints and master shards that are hosted on the failed node."

ה-Cluster Manager מנותק מ-Redis לחלוטין כדי לא להשפיע על ביצועים, ואחראי גם על איתור הכשל וגם על ביצוע ה-failover.

3. Quorum-based decisions

הקלאסטר דורש quorum (מספר אי-זוגי של nodes — 3, 5, 7) לקבלת החלטות אדמיניות. ב-5 nodes ניתן לעמוד בנפילת 2 nodes ועדיין לשמור quorum.

4. Rack-Zone Awareness

אופציונלי. מבטיח ש-master ו-replica של אותו shard לא רק על nodes שונים, אלא על racks/AZs שונים. עמידות בנפילת AZ שלם בענן.

⚡ סימולציית אסונות — אינטראקטיבית בתצוגה למעלה

איך להפעיל:
  ✕ הפלת node — Hover על כל node בתצוגה ולחץ על העיגול האדום ✕ שמופיע בפינה (לחיצה שנייה משחזרת).
  ↻ Failover shard — לחץ על master shard (M1, M2...) → במודאל לחץ "Failover".
  ✕ מחיקת shard — לחץ על כל shard → במודאל לחץ "Delete shard".

BDB Architecture — בחר את הארכיטקטורה

ל-BDB (Back-end Database — שם פנימי של database ב-Redis Enterprise, נחשף דרך REST API ב-/v1/bdbs/) יש בחירות ארכיטקטורה מרובות. הנה הבחירות העיקריות, עם תצוגה ויזואלית שמתעדכנת לפי הבחירה.

4. Synchronization Mode — Centralized vs Distributed

ב-Active-Active databases יש 2 מצבי synchronization mode שקובעים איך תעבורת הסנכרון בין הקלאסטרים מטופלת. לפי Configure distributed synchronization — docs.redis.io:

הבדל חשוב — synchronization mode vs sync types:
  • Synchronization mode (Centralized/Distributed) — איך תעבורת הסנכרון מנותבת בין proxy endpoints.
  • Sync types (Partial/Full) — איך הנתונים מועברים בכל סשן sync:
    • Partial sync — ה-backlog של פעולות מאז offset מועבר כ-raw operations.
    • Full sync — כל ה-data מועבר כקובץ RDB, ואחר כך partial sync.

שירותים ופרוססים — Per Node

להלן כל השירותים שרצים על נוד Redis Enterprise. רובם מנוהלים ע"י supervisord. חלקם רצים פעם אחת לכל נוד (singleton), וחלקם מופע נוסף לכל DB/shard.

a. supervisord

תפקיד: Process supervisor — השורש של כל הפרוססים על הנוד. אחראי לאתחל, לעצור, ולהפעיל מחדש שירותים שנופלים. הוא PID 1 בתוך הקונטיינר/דמון של Redis Enterprise.

קונפיג: /etc/opt/redislabs/supervisord.conf

פקודות שימושיות:

# רשימת כל השירותים
sudo supervisorctl status

# הפעלה מחדש של שירות
sudo supervisorctl restart dmc_proxy

# הפעלה מחדש של כל הסטאק על הנוד
sudo /opt/redislabs/sbin/services restart

b. stats_archiver

תפקיד: אוסף מטריקות מכל ה-shards ושירותים על הנוד, כולל לטנסי, throughput, memory usage, hit/miss ratio, ושומר אותן ב-time-series database פנימי. ה-data משמש את ה-UI ואת ה-REST API להצגת graphs.

פורטים: מאזין פנימית — לא חשוף לרשת חיצונית. ה-UI ניגש דרך ה-REST API (פורט 9443).

תהליכי בנים: בדרך כלל ילד יחיד של supervisord, עם threads פנימיים לאיסוף.

נתונים: שמור ב-/var/opt/redislabs/persist/stats/

c. Cluster Manager (failure detection)

לפי Terminology, ה-Cluster Manager אחראי על איתור כשלים וביצוע failover. ציטוט: "The cluster manager is designed in a way that is totally decoupled from the Redis operation. This enables RS to react in a much faster and accurate manner to failure events, so that, for example, a node failure event triggers mass failover operations of all the master endpoints and master shards that are hosted on the failed node."

תפקידי ה-Cluster Manager לפי התיעוד:

  • הקצאת shards (איפה הם נוצרים)
  • החלטה מתי shards מועברים
  • ניטור גודל database ו-endpoints על כל הnodes
  • הפעלת תהליכי resharding, provisioning, de-provisioning
  • איסוף סטטיסטיקות אופרטיביות
  • אכיפת רישוי

ה-Cluster Manager בנוי בארכיטקטורה shared-nothing — ללא תלות ב-Redis ובלי השפעה על ביצועיו.

d. Discovery Service (תאימות Sentinel)

Redis Enterprise חושף Discovery Service על port 8001 (TCP) שמספק תאימות ל-Sentinel API לקליינטים שמשתמשים בפרוטוקול Sentinel discovery.

ה-Discovery Service מאפשר לקליינטים לקבל את ה-topology העדכנית של ה-cluster (אילו nodes פעילים, איפה ה-master shard) — בדומה ל-Sentinel ב-OSS Redis, אבל ב-Enterprise זה endpoint שמחזיר את המידע, ולא תהליך פעיל של failover.

ה-failover עצמו ב-Enterprise מנוהל ע"י Cluster Manager (ראה למעלה), לא דרך ה-Discovery Service.

מקור: Discovery service — docs.redis.io

e. Envoy (gossip_envoy)

תפקיד: service mesh / proxy ל-internal gossip protocol. מטפל בתעבורת ה-cluster control-plane בין הנודים — שינויי קונפיג, פרסום מטריקות, gossip.

פורטים: 9081 (admin), בנוסף לפורטי gossip 3333-3339.

הערה חשובה: ב-versions חדשים יותר של Redis Enterprise נכנס Envoy כמרכיב מרכזי לניהול תעבורה פנים-קלאסטרית. הוא מאפשר security (mTLS), telemetry, והוא הבסיס למיגרציה ל-Service Mesh.

f. DMC Proxy (dmc_proxy)

תפקיד: Dynamic Multi-tenant Connection proxy. נקודת הכניסה האמיתית של קליינטים. רץ על כל נוד. בעת התחברות של קליינט:

  1. פותח חיבור TCP אחד מהקליינט
  2. מחשב hash slot ל-key
  3. שואל את CCS באיזה shard ה-slot
  4. פותח/משתמש ב-connection pool ל-shard המתאים
  5. מבצע multiplexing: עשרות אלפי קליינטים → מאות חיבורים פנימיים

תהליכי בנים: ה-dmc_proxy הוא single-process עם threads רבים (ולא child processes). מספר ה-threads קונפיגורבילי per-DB.

פורט: 1968 (admin פנימי), פורטי DB דינמיים (10000-19999) הם ב-listening mode של ה-proxy.

g. crdb_coordinator

תפקיד: מתאם בין-קלאסטרי לכל CRDB. רץ פעם אחת לכל קלאסטר (singleton), על אחד הנודים. אחראי על:

  • קונפיגורציה של ה-CRDB (אילו קלאסטרים משתתפים, מה ה-instance ID של כל אחד)
  • תיאום היצירה והעדכון של ה-CRDB
  • ניהול מצב סנכרון גלובלי
  • פתרון קונפליקטים ברמת ה-metadata

תקשורת: מדבר עם ה-coordinators של הקלאסטרים האחרים דרך פורטי REST API (9443) ופורטי tcp ייעודיים ל-CRDB.

פורט: סביב 9081 ו-9090-9099 (משתנה לפי גרסה).

h. cnm_exec (Cluster Node Manager Executor)

תפקיד: הסוכן על כל נוד שמבצע פקודות מהקלאסטר. כשמשתמש לוחץ "Add DB" ב-UI:

  1. הבקשה מגיעה ל-REST API → cm_server
  2. cm_server מעדכן את CCS
  3. cnm_exec על כל נוד רואה את השינוי ומבצע פעולות לוקאליות:
    • spawn של redis-server חדש (shard)
    • קונפיגורציה של ה-DMC Proxy
    • הקצאת port חדש
    • הפעלת replication אם צריך

תהליכי בנים: cnm_exec מבצע fork+exec לתהליכי redis-server. אלו מופיעים תחתיו בעץ הפרוססים, אך לאחר הפעלה הם תחת supervisord.

שירותים נוספים על הנוד

ccs (Cluster Configuration Store)

Distributed KV store שמחזיק את כל מצב הקלאסטר (אילו DBs, אילו shards, איפה כל אחד יושב). מבוסס על אלגוריתם quorum (Paxos-like). רץ על כל נוד ומסונכרן בין כולם. ה-leader נבחר דינמית.

פורט: 8002, 9443 (לקריאה דרך REST).

pdns_server (Private DNS)

שרת DNS פנימי שמתחזק את ה-FQDNs של ה-DBs בקלאסטר. מאפשר לקליינטים לפנות לפי שם (redis-12000.cluster.local) ולקבל IP של נוד שזמין.

פורט: 53/UDP+TCP, 5353 (mDNS).

mdns_server

Multicast DNS — מאפשר discovery של נודים בתוך LAN ללא DNS חיצוני. שימושי בעיקר באתחול קלאסטר.

פורט: 5353/UDP multicast.

alert_mgr

Alert Manager — מקבל events מ-stats_archiver ומ-cnm_exec, ומחליט אם להעלות alert (לדוגמה: shard down, memory > 80%, sync_lag > 5s). שולח ל-syslog/email/Prometheus alertmanager.

cm_server (Cluster Manager Server)

מספק את ה-REST API של הקלאסטר ואת ה-Web UI. רץ על כל נוד אך מטפל בבקשות בתורנות (active-active LB דרך FQDN של הקלאסטר).

פורטים: 8443 (UI), 9443 (REST API).

heartbeatd / node_wd

Heartbeat protocol בין הנודים. כל נוד שולח heartbeat לאחרים כל ~250ms. אם 3 חמיצות רצופות — הנוד נחשב down.

פורטים: 3333, 3334, 3335 (gossip+heartbeat).

crdb_syncer

נושא טעון! רץ פעם לכל CRDB-instance, על נוד אחד בקלאסטר. אחראי על קריאת ה-replication-stream של ה-CRDB ושליחתו לקלאסטרים האחרים. ראה טאב CRDB Syncer לפרטים מלאים.

פורט: 20000-29999 (סינכרון בין-קלאסטרי).

redis-server (per shard)

הפרוסס של Redis עצמו — אחד לכל shard! אם יש 5 master shards + 5 replicas → 10 פרוססים של redis-server מפוזרים בין הנודים. כל אחד עם פורט משלו.

פורט: 10000-19999 (דינמי, מוקצה ע"י cnm_exec).

shard_listener / replica_link

חוט פנימי בתוך redis-server שמטפל ב-replication. כל replica יוצר חיבור TCP ל-master ומקבל replication stream באופן רציף.

טיפ ללמידה: אם תרצה לראות את כל זה רץ על מכונה אמיתית, התקן Redis Enterprise (יש docker image: redislabs/redis), הרם 3-5 מופעים, אתחל קלאסטר, ואז הרץ ps auxf בתוך הקונטיינר. תראה את כל ה-tree.

איך עובד הסנכרון של CRDB

  1. קליינט בארה"ב מבצע SET user:42 "Alice" ל-Cluster A.
  2. ה-write נכתב ל-master shard המתאים ב-A — מקבל ack מיידי.
  3. בנפרד ובאסינכרון, ה-master רושם ב-replication-stream פנימי.
  4. crdb_syncer ב-Cluster A קורא את ה-stream הזה.
  5. ה-syncer ממיר את ה-event לפורמט CRDB (כולל causal-context, vector clocks).
  6. ה-syncer שולח את ה-event דרך פורט 22001 ל-Cluster B (TLS אופציונלי).
  7. ה-syncer ב-Cluster B מקבל, מפענח, ומחיל על ה-master shard המקביל ב-B.
  8. אם בו זמנית היה write גם ב-B על אותו key — CRDB מבצע merge אוטומטי לפי סוג ה-data type.

פתרון קונפליקטים — CRDTs

Redis Enterprise תומך ב-CRDTs לסוגי הנתונים הבאים:

  • String — Last-Write-Wins (LWW) באמצעות vector clocks
  • Hash — field-level merge
  • List — OR-set semantics
  • Set — union (add wins over remove)
  • Sorted Set — score = sum across replicas
  • Counter — true PN-Counter (positive-negative counter)
  • Stream — append-only, נשמר order לפי vector clock

המשמעות: לעולם לא יקרה אובדן כתיבה גם בלי קואורדינציה גלובלית בעת ה-write.

בתוך הקלאסטר — סנכרון בין master shard ל-replica shard:

  • async ברירת מחדל — ה-master מאשר את ה-write לקליינט מיד, ושולח את הdiff ל-replica ברקע.
  • sync WAIT-based — קליינט שולח אחרי ה-SET את הפקודה WAIT 1 100 — מחכה עד ש-1 replica קיבל את הכתיבה (עד 100ms timeout). מבטיח durability גם מול אובדן master.

בין הקלאסטרים — Replica-Of ו-CRDB:

  • תמיד אסינכרוני. אין דרך לקבל גם low-latency וגם cross-region sync.
  • זה סטנדרט תעשייתי לאיכות הזמינות מול CAP theorem.

מה רואים ב-UI אחרי כיבוי?

  • במסך CRDB list — סטטוס משתנה ל-stopped או paused (תלוי בשיטה).
  • Sync lag מתחיל לעלות ולא מתאפס. אחרי כמה דקות ה-UI יציג alert "CRDB sync lag exceeded threshold".
  • Replication backlog במאסטרים מתחיל לתפוח (RAM!) — אם הוא עובר את ה-replication-backlog-size תהיה bgsave מלאה במקום differential resync כשהsyncer יחזור.
  • Status icon של ה-CRDB משתנה לאדום/צהוב.
  • Per-cluster connection state מציג "disconnected" לקלאסטר שכובה.
  • Writes נמשכים ב-cluster A — אבל לא מגיעים ל-B עד שה-syncer יחזור.
  • אם תפעיל מחדש — ה-syncer יבצע partial resync (אם backlog זמין) או full resync (RDB → cluster B).

הבעיה הקריטית — Master Shard לא יושב על אותו Node כמו ה-Syncer

זה תרחיש שצריך להבין לעומק כי הוא משפיע על ביצועים ויציבות.

תיאור המצב

ה-crdb_syncer רץ על נוד אחד (לדוגמה node-3). ה-master shard M1 של ה-CRDB יושב על נוד אחר (לדוגמה node-1). כל פעם שצריך לקרוא את ה-replication stream של M1, ה-syncer פותח חיבור TCP מ-node-3 ל-node-1.

הנזק שיכול להיגרם

  • Latency נוסף ברשת — כל event עובר בנוסף ב-network hop פנימי. בדרך כלל זניח (sub-ms בתוך rack), אבל בקלאסטרים שמפוזרים על AZ-ים שונים זה כבר משמעותי.
  • תעבורה כפולה ברשת — ה-stream גם נשלח ל-replica של אותו shard וגם ל-syncer. אם ה-syncer לא על נוד ה-master, כל ה-replication stream עובר פעמיים ברשת (פעם ל-replica, פעם ל-syncer).
  • נקודת כשל נוספת — אם הרשת בין node-3 ל-node-1 נכשלת, ה-syncer לא מקבל events גם אם המאסטר עצמו בריא לחלוטין. זה מייצר sync_lag לא הכרחי.
  • בעיית CPU על נוד הסינקר — ה-syncer צריך לקרוא TCP, לפענח, להמיר ל-CRDB format. אם הוא רחוק מהמאסטר, ה-buffer ב-memory צומח, ולחץ על ה-CPU גובר.
  • Failover מחמיר את הבעיה — אם ה-master עובר ל-replica שלו (failover), המיקום משתנה. אם ה-syncer לא הוגדר לעקוב אחר ה-master החדש (re-bind) — יש קפיצה זמנית של תעבורה.

HA & Failover

איך הקלאסטר מחזיק את עצמו פועל גם כשנופל נוד או shard.

תרחישי כשל וההתנהגות

נפילת replica shard

אין השפעה על writes/reads. ה-cnm_exec יזהה את הנפילה (timeout על ה-replication-link), ויפעיל אותו מחדש או ייצור replica חדש על נוד אחר.

נפילת master shard

  • Sentinel מזהה תוך ~5 שניות (ping-failures).
  • בוחר את ה-replica העדכני ביותר ומקדם אותו ל-master.
  • מעדכן את DMC Proxy על המיפוי החדש.
  • קליינטים מקבלים שגיאה זמנית (1-3 שניות), ואז ממשיכים לעבוד מול ה-master החדש.
  • שגיאה זמנית כי ה-DMC Proxy מנתב מחדש בצורה שקופה.

נפילת node שלם

  • כל ה-shards שעליו (master + replica) נופלים יחד.
  • Master shards שהיו עליו: ה-replicas שלהם (על נודים אחרים) מקודמים.
  • Replica shards שהיו עליו: ייווצרו replicas חדשות על נודים אחרים אם יש מקום.
  • הקלאסטר נשאר עם 4 נודים — quorum נשמר (3/5).

נפילת 2 נודים

הקלאסטר עדיין עם 3 נודים = quorum. עובד אבל בלי redundancy מלא.

נפילת 3 נודים

אבד quorum. הקלאסטר עובר ל-read-only. צריך לפתור את הנפילה ולעלות לפחות נוד אחד מהנפלים.

קלאסטר Wide

1. כיצד בנויה ארכיטקטורת קלאסטר ברדיס Enterprise? כיצד ממומש HA?

קלאסטר Redis Enterprise מורכב מ-3 או יותר nodes (מומלץ מספר אי-זוגי: 3, 5, 7). הארכיטקטורה סימטרית — כל node מריץ את אותם רכיבים.

4 הרכיבים הרשמיים בכל node (לפי Terminology):

  • Node — המכונה עצמה (פיזית/וירטואלית/קונטיינר)
  • Zero-latency Proxy — Proxy אחד לכל node, multi-threaded ב-C, cut-through architecture
  • Cluster Manager — מנהל את ה-node (וב-master node — את הקלאסטר כולו)
  • Shards (Redis instances) — מספר Redis processes לכל node

HA ממומש כך:

  1. Master/Replica Shard — כל master shard עם replica על node שונה. סנכרון אסינכרוני (ניתן לאכוף sync עם WAIT).
  2. Cluster Manager מאתר node failure event ומפעיל mass failover אוטומטית.
  3. Master/replica לעולם לא על אותו node (חוק הזהב של Redis Enterprise).
  4. Rack-Zone awareness אופציונלי — לעמידות גם בנפילת AZ שלם.
  5. ל-DR מלא בין אזורים: Replica-Of (Active-Passive) או Active-Active (CRDB).

3. אילו רכיבים רצים על כל node?

לפי Terminology — docs.redis.io, הרכיבים הרשמיים הם:

Zero-latency Proxy

ציטוט: "Each node includes one zero-latency, multi-threaded proxy (written in low-level C) that masks the underlying system complexity. The proxy oversees forwarding Redis operations to the database shards on behalf of a Redis client."

בנוי בארכיטקטורת cut-through, משתמש ב-instruction pipelining גם אם הקליינט לא ביקש זאת. ניתן להגדיר proxy policies (single / all-master-shards / all-nodes).

Cluster Manager

ציטוט: "The cluster manager oversees all node management-related tasks, and the cluster manager in the master node looks after all the cluster related tasks."

תפקידיו (לפי התיעוד הרשמי): הקצאת shards, החלטה מתי להעביר shards, ניטור גודל database, ניטור endpoints, הפעלת resharding/provisioning, איסוף סטטיסטיקות, אכיפת רישוי.

בנוי בארכיטקטורה shared-nothing — מנותק לחלוטין מ-Redis כדי לא להשפיע על throughput/latency.

Shards (Redis instances)

כל node הוא container ל-multiple Redis instances. כל shard הוא תהליך redis-server בודד, עם פורט משלו. shards יכולים להיות master או replica של אותו database.

תהליכי implementation שניתן לראות ב-ps

ברמה הטכנית מתחת למכסה המנוע, ה-Cluster Manager וה-Proxy מורכבים מתהליכים כמו: supervisord (process supervisor), cnm_exec (cluster node manager executor), ccs (cluster config store), dmc_proxy (proxy implementation), stats_archiver, pdns_server, mdns_server, alert_mgr, crdb_coordinator, crdb_syncer. אלה implementation details — התיעוד הציבורי משתמש במונחים High-level: Cluster Manager + Proxy.

4. מה ההבדל בין Master Shard ל-Replica Shard? ולמה לא יושבים על אותו node?

Master shard — מקבל את כל ה-writes לאותם slots, ומבצע replication ל-replicas שלו.

Replica shard — מסונכרן עם ה-master. read-only. במקרה של נפילת master, מקודם.

למה לא על אותו node? אם node יחיד נופל ויש עליו את ה-master + ה-replica של אותו shard — אבדנו את הנתון לחלוטין (אין לאן לעשות failover). הקלאסטר מאכלס אותם דווקא על נודים שונים, ובאופן אידיאלי על racks/AZs שונים.

5. מה זה ה-Proxy ולמה הוא נחוץ?

לפי Terminology, השם הרשמי הוא "zero-latency proxy". ציטוט:

"Each node includes one zero-latency, multi-threaded proxy (written in low-level C) that masks the underlying system complexity. The proxy oversees forwarding Redis operations to the database shards on behalf of a Redis client."

תכונות עיקריות:

  • One per node — Proxy אחד לכל node בקלאסטר. ה-policy (single / all-master-shards / all-nodes) קובע אילו proxies bound לאיזה database.
  • Multi-threaded ב-C — ביצועים גבוהים, multi-core utilization.
  • Cut-through architecture — מעביר פעולות ישירות ל-shard המתאים בלי buffering מיותר.
  • Instruction pipelining — ציטוט: "the proxy might use instruction pipelining even if not instructed to do so by the client" — אופטימיזציה אוטומטית.
  • Standard Redis client compatibility — מסתיר את מורכבות הקלאסטר; הקליינט עובד כאילו זה Redis בודד.

בקוד הפנימי השם הוא dmc_proxy. בתיעוד הציבורי המונח הוא "Proxy" / "zero-latency proxy".

6. איך עובד quorum הקלאסטר? מה קורה כשהוא מאבד quorum?

הקלאסטר דורש מספר אי-זוגי של nodes (3, 5, 7) כדי לקבל החלטות אדמיניות בצורה דטרמיניסטית.

חישוב quorum: נדרש ⌊N/2⌋+1 nodes פעילים — דהיינו רוב.

  • קלאסטר 3 nodes → quorum = 2 → סובל נפילה של node אחד.
  • קלאסטר 5 nodes → quorum = 3 → סובל נפילה של 2 nodes.
  • קלאסטר 7 nodes → quorum = 4 → סובל נפילה של 3 nodes.

אובדן quorum (פחות מהמינימום הנדרש):

  • הקלאסטר עובר למצב מוגבל — ה-Cluster Manager לא יכול לקבל החלטות אדמיניות.
  • פעולות ניהול (יצירת DB, failover מתוכנן, migrate) נחסמות.
  • databases פעילים ממשיכים לעבוד אבל בלי resilience לכשלים נוספים.
  • הפתרון: להחזיר את ה-nodes שנפלו או להפעיל recovery process.

הסיבה ל-מספר אי-זוגי: למנוע tie במצב של network partition (split-brain). עם מספר זוגי הקלאסטר עלול להתפצל לשתי קבוצות שוות שאף אחת מהן לא יכולה לקבל החלטות.

7. מה ההבדל בין סינכרון אסינכרוני לסינכרוני בתוך הקלאסטר?

אסינכרוני (ברירת מחדל): ה-master מאשר את ה-write לקליינט מיד אחרי שכתב לזיכרון שלו. הוא שולח את ה-diff ל-replica ברקע. async

WAIT-based (semi-sync): הקליינט שולח אחרי ה-SET את WAIT 1 100 — מחכה עד ש-1 replica קיבל את הכתיבה (מקסימום 100ms). אם לא הגיע — מקבל back את מספר ה-replicas שאישרו. sync

בין הקלאסטרים (CRDB / Replica-Of) — תמיד אסינכרוני.

אבל! AOF fsync יכול להיות סינכרוני (אופציה always) לדיסק — לא ל-replica.

8. כיצד נראית הארכיטקטורה של ה-CRDB syncer?

ה-syncer הוא process עם 3 threads עיקריים: INPUT, TRANSFORM, OUTPUT.

  1. INPUT — פותח חיבור TCP אל ה-master shard המקומי, רושם PSYNC, ומקבל replication-stream.
  2. TRANSFORM — ממיר כל פקודה לפורמט CRDB עם vector clocks ו-causal context.
  3. OUTPUT — שולח את ה-events דרך TLS לקלאסטרים האחרים על פורט 22001+. מנהל גם backlog למקרה של disconnection זמני.

בצד הקבלה: ה-syncer של הקלאסטר השני מקבל, מפענח, מבצע CRDT merge, ושולח ל-master shard המקביל.

הסבר ויזואלי מלא: ראה טאב CRDB Syncer.

מה יראה ה-UI:

  • גרף ה-sync lag יתחיל לעלות ולא יתאזן.
  • אינדיקטור של "X clusters connected" יראה פחות מהמספר המוגדר.

10. גיליתי ש-master shard לא יושב ביחד עם ה-syncer על אותו node, מדוע זו יכולה להיות בעיה?

ה-syncer צריך לקרוא את ה-replication-stream של כל master shard המשתתף ב-CRDB. אם הוא לא על אותו node — זה דורש חיבור TCP נוסף ברשת הפנימית של הקלאסטר.

הנזקים האפשריים:

  • Latency נוסף — בכל hop ברשת. ב-LAN זה sub-ms, אבל ב-AZ-multi זה כבר 1-5ms.
  • תעבורה כפולה ברשת — ה-stream נשלח גם ל-replica וגם ל-syncer. אם ה-syncer לא local, ה-stream עובר את הרשת פעמיים.
  • Bottleneck על נוד הסינקר — אם הוא צריך לאסוף streams מ-3-4 master shards שמפוזרים על נודים אחרים, ה-NIC שלו יכול להיהפך bottleneck.
  • נקודת כשל נוספת — נפילה של הקישור node→node ספציפי תפסיק את הסנכרון גם אם ה-master עצמו בריא.
  • צריכת CPU גבוהה יותר על נוד הסינקר בגלל buffering ו-parsing נוספים.
  • בעיית failover — אם ה-master עובר ל-replica, ה-syncer צריך לפתוח חיבור חדש לנוד החדש (re-bind), וזה גורם ל-spike של lag.

איך פותרים:

# 1. אופציה א: להזיז את ה-master shard לנוד עם הsyncer
sudo /opt/redislabs/bin/rladmin migrate shard <shard-id> \
  target_node <node-with-syncer>

# 2. אופציה ב: להפעיל Distributed synchronization mode
# כדי לפזר את תעבורת ה-sync על כל proxy endpoint זמין
# (מצריך proxy policy של all-master-shards או all-nodes)
sudo /opt/redislabs/bin/rladmin bind db <db-name> \
  endpoint <endpoint-id> policy all-master-shards

# 3. אופציה ג: שינוי shard placement policy
sudo /opt/redislabs/bin/rladmin tune cluster \
  default_shards_placement dense

בדיקה אחרי הפתרון: crdb-cli task list ו-crdb-cli crdb get --crdb-guid <guid> — לוודא שמצב הסנכרון תקין.

11. מה ההבדל בין Active-Active ל-Active-Passive?

Active-Active (CRDB):

  • שני (או יותר) קלאסטרים writable.
  • סנכרון bidirectional אסינכרוני.
  • פתרון קונפליקטים אוטומטי דרך CRDTs.
  • שימוש: אפליקציות גלובליות, low-latency writes לכל אזור.

Active-Passive (Replica-Of):

  • קלאסטר אחד primary writable, השני secondary read-only.
  • סנכרון unidirectional אסינכרוני.
  • אין פתרון קונפליקטים (אין כתיבות מקבילות).
  • שימוש: DR, read-replicas גלובליים, BI workloads.
  • במקרה של disaster — promotion ידני של secondary ל-primary חדש.
12. כיצד מתבצע failover אוטומטי? כמה זמן זה לוקח?

אומת מהתיעוד: "The entire failover process occurs in single-digit seconds, without manual intervention. In a quiet network environment, a Redis Enterprise cluster can be easily tuned to support a constant single-digit (<10 sec) failover time" — Highly available Redis.

אז: failover < 10 שניות באופן ודאי בסביבה רגועה, לאחר tuning. ב-cloud רועש זה יכול להיות קצת יותר.

שלבי ה-failover (לפי הבנה כללית — לא כל הפרטים מתועדים פומבית!)

  1. T+0: ה-master נופל (kernel panic / OOM / network).
  2. T+~5s: Node Watchdog ו/או Cluster Watchdog מאתרים את הכשל. ערך ברירת מחדל של down-after-milliseconds ב-Sentinel היה 5000ms — ב-Enterprise התיזמון הספציפי לא מתועד פומבית.
  3. T+~5-7s: Cluster Manager מקבל את ה-event ויוזם failover דרך CCS.
  4. T+~7-9s: Replica מקודם ל-master, DMC Proxies מתעדכנים על routing חדש.
  5. T+<10s: קליינטים ממשיכים לקבל replies.

הערה: השלבים המדויקים והעיתויים לא מפורטים בתיעוד הציבורי של Redis. הביטחון היחיד: <10 שניות בסביבה רגועה. בזמן ה-failover הקליינטים מקבלים connection errors זמניים — כדאי retry.

13. מה זה shard? כמה shards לכל BDB?

לפי Terminology, ה-BDB יכול להיות באחד מ-4 מודלים רשמיים:

  1. Standard Redis Database — "A single Redis shard with no replication or clustering."
  2. Highly Available Redis Database — "Every database master shard has a replica shard, so that if the master shard fails the cluster can automatically fail over to the replica with minimal impact. Master and replica shards are always placed on separate nodes to ensure high availability."
  3. Clustered Redis Database — "The data stored in the database is split across several shards. The number of shards can be defined by the user. Various performance optimization algorithms define where shards are placed within the cluster. During the lifetime of the cluster, these algorithms might migrate a shard between nodes."
  4. Clustered and Highly Available Redis Database — "Each master shard in the clustered database has a replica shard, enabling failover if the master shard fails."

Shard = Redis instance בודד (תהליך redis-server). יחלוקת keys בין shards מבוססת על hash slots (16384 slots) של מודל Redis Cluster.

פיזור ה-shards בין nodes נקבע ע"י shard placement policy (dense או sparse) — ראה טאב BDB Architecture.

17. מה זה Eviction Policy ואיך מגדירים אותו ב-CRDB?

לפי Eviction policy — docs.redis.io:

1. ברירת המחדל ל-Active-Active databases

ברירת המחדל היא noeviction — אין eviction בכלל. ה-server יחזיר error על כל פקודת write שתנסה ליצור keys חדשים, ב-DB שמלא.

2. גרסאות שתומכות ב-eviction policies ב-AA

  • לפני Redis Software 6.0.20 — noeviction היה היחיד הנתמך ל-AA databases.
  • מ-Redis Software 6.0.20 ואילך — כל ה-eviction policies נתמכים ל-AA, פרט למקרה אחד: כש-Auto Tiering (Redis Flex) מופעל, אז eviction policies אינם נתמכים ל-AA.

3. רשימת ה-policies (כשנתמכים)

  • noeviction — ברירת מחדל ל-AA. אין מחיקה, errors על writes חדשים.
  • allkeys-lru / volatile-lru
  • allkeys-lfu / volatile-lfu
  • allkeys-random / volatile-random
  • volatile-ttl

4. התנהגות מיוחדת ב-AA

  • ה-eviction מתחיל ב-80% memory limit — מוקדם יותר מ-standalone — כי הוא דורש propagation לכל הקלאסטרים המשתתפים.
  • במקרה של בעיות רשת בין ה-AA instances: זיכרון משוחרר רק כשכל ה-instances מסונכרנים. אם אין תקשורת בין הקלאסטרים, יכול להיווצר OOM כי ה-eviction לא יכול להתפשט.

שורה תחתונה: ב-AA הברירת מחדל היא noeviction, ובסביבה מודרנית ניתן לבחור גם policies אחרות (חוץ מאשר עם Auto Tiering).

18. מה ההבדל בין AOF ל-RDB persistence?

אומת מהתיעוד — Configure database persistence:

אופציות ה-Snapshot ב-Redis Enterprise UI/REST

  • Snapshot every 1 hour — backup מלא כל שעה
  • Snapshot every 6 hours — backup מלא כל 6 שעות
  • Snapshot every 12 hours — backup מלא כל 12 שעות
  • None — ללא persistence

אופציות AOF (לdurability גבוהה יותר)

  • AOF — fsync every 1 sec — Redis יעשה fsync לכל write חדש כל שנייה. אובדן מקסימלי בעת תקלה: ~1 שנייה.
  • (ב-OSS יש גם always ו-no, אבל ב-Enterprise UI הם לא מוצגים בצורה זו.)

הבדל המושגי

RDB = point-in-time snapshot, מהיר ל-recovery, חסכון בזיכרון. AOF = append-only log של כל פעולה, durability גבוהה יותר, recovery איטי יותר. ב-Redis Enterprise אפשר להפעיל שניהם ביחד (Hybrid) — AOF ל-durability ב-runtime, RDB ל-backup periodic.

19. איך מבצעים upgrade של הקלאסטר ללא downtime?

לפי Upgrade a Redis Software cluster:

שתי שיטות upgrade רשמיות

  • In-place upgrade — שדרוג גרסה על הnodes הקיימים, אחד אחרי השני. הכי נפוץ ל-software upgrades.
  • Rolling upgrade (replace_node) — מוסיפים nodes חדשים עם הגרסה החדשה, ומחליפים את הקיימים. שימושי גם ל-OS upgrade ולשינוי תשתית.

קונספט קריטי: ה-Cluster Master Node

למרות שהnodes שווים פונקציונלית, יש node אחד שנושא את ה-cluster master role (CCS leader / cluster-control-plane primary). חייב לשדרג אותו ראשון ברוב המקרים — אחרת הקלאסטר עלול להיתקע או ה-join יכשל.

# איתור ה-cluster master
sudo /opt/redislabs/bin/rladmin status nodes

# החלפת ה-cluster master ידנית (אם צריך)
sudo /opt/redislabs/bin/rladmin cluster master set <node-id>

תהליך In-place Upgrade (פישוט)

  1. שדרוג ה-cluster master node ראשון
  2. הרצת ה-installer ב-node (נשען על השלים אוטומטית): sudo ./install.sh -y
  3. השלמת ה-upgrade ב-node ו-RIfresh של השירותים
  4. חוזרים על התהליך ב-node הבא
  5. בסוף: שדרוג ה-DBs (modules / version): rladmin upgrade db <db-name>

תהליך Rolling Upgrade (replace_node)

# מוסיפים node חדש עם גרסה חדשה
# שמחליף node ישן
rladmin cluster join nodes <cluster_member_ip> \
  username <username> \
  password <password> \
  replace_node <old_node_id>

הזהירות החשובה

  • חריג: אם הקלאסטר מאחורי load balancer (כמו F5), יתכן שלא ניתן לשדרג את ה-cluster master ראשון. צריך להתייעץ עם עוד תיעוד אם זה המקרה.
  • בזמן upgrade — אסור לבצע פעולות אדמיניות (יצירת DB, failover מתוכנן). זה עלול לגרום לתוצאות לא צפויות.
  • זמן: תלוי בגודל ה-data. ה-shards עוברים בין nodes (drain), זה הצוואר בקבוק.

שורה תחתונה לבחינה: אל תאמר "node drain + package upgrade". אמור: "upgrade מתבצע ע"י rladmin upgrade; קודם ה-cluster master node, אחר כך השאר; שתי שיטות — in-place או rolling עם replace_node".

22. מהן 3 ה-Proxy Policies של DMC ואיזה לבחור?

על פי תיעוד רשמי של Redis (Configure proxy policy):

  1. Single (ברירת מחדל) — פרוקסי אחד bound ל-BDB. נמוך משאבים. אם יש פעולות רבות אך לא traffic קיצוני — מתאים. הכי פשוט.
  2. All Primary Shards (REST: all-master-shards) — פרוקסי על כל node שמארח master shard. latency נמוך יותר. מתאים לרוב מקרי הuse case שדורשים פרוקסים מרובים, במיוחד עם OSS Cluster API.
  3. All Nodes — פרוקסי על כל nodes בקלאסטר. שימושי כשיש load balancer חיצוני, או environment ללא DNS אמין. צריכת משאבים גבוהה יותר ויותר latency פוטנציאלי.

ערכי REST API: "proxy_policy": "single | all-master-shards | all-nodes"

איך לבחור?

  • אם traffic רגיל וקליינט אחד שמתחבר ל-FQDN → single
  • אם רוצים latency הכי נמוך אפשר עם הרבה shards → all-master-shards
  • אם יש LB מול הקלאסטר → all-nodes
23. מהן 2 ה-Shard Placement Policies?

על פי Shard placement policy:

  1. Dense — מרכז את ה-shards על כמה שפחות nodes. כש-node מתמלא, חלק מועברים ל-node אחר. מומלץ ל-Redis on RAM ללא OSS Cluster API.
  2. Sparse — מפזר shards על כמה שיותר nodes. מומלץ ל-Redis on RAM עם OSS Cluster API ול-Auto Tiering (Flash).

לדוגמה, BDB עם 3 master + 3 replica = 6 shards בקלאסטר 5-nodes:

  • Dense: 3 masters על node-1, 3 replicas על node-2, nodes 3-5 פנויים
  • Sparse: shard אחד או שניים בכל node, מפוזר על 5 nodes

בחירה נשמרת ב-CCS וניתן לעדכן: rladmin tune cluster default_shards_placement dense | sparse

24. איך נעשה Failover ב-Redis Enterprise?

ה-failover ב-Redis Enterprise מנוהל ע"י Cluster Manager. ציטוט מהתיעוד הרשמי (Terminology):

"The cluster manager is designed in a way that is totally decoupled from the Redis operation. This enables RS to react in a much faster and accurate manner to failure events, so that, for example, a node failure event triggers mass failover operations of all the master endpoints and master shards that are hosted on the failed node."

תהליך ה-failover:

  1. Cluster Manager מאתר node failure event (מנגנוני ה-monitoring הפנימיים).
  2. מפעיל mass failover: כל ה-master shards שעל הנוד הנפול מקבלים replicas מקודמים ל-master.
  3. ה-master endpoints מעודכנים — Proxies מתחילים לנתב לnodes החדשים.

לגבי Sentinel: ה-Discovery Service (port 8001) חושף API תואם Sentinel לקליינטים. אבל ה-failover עצמו מנוהל ע"י Cluster Manager — לא ע"י תהליך Sentinel.

25. מה ההבדל בין partial sync ל-full sync ב-syncer?

על פי Syncer process:

Partial sync: ה-backlog של פעולות מאז offset מסוים מועבר כ-raw operations (פעולות ב-RESP). זה מהיר ויעיל. קורה כש:

  • ה-replica/syncer מחובר רוב הזמן וקיבל כל ה-stream באופן רציף
  • היה disconnect קצר וה-backlog עדיין מכיל את כל מה שהוחמץ

Full sync: כל ה-data של ה-master מועבר ל-replica כקובץ RDB, ואחר כך partial sync ממשיך מנקודת ה-RDB. זה איטי וחלוקת משאבים. קורה כש:

  • ה-replica/syncer חדש לחלוטין (אין נתונים)
  • היה disconnect ארוך מדי וה-backlog "התקפל" (ה-offset הישן כבר לא קיים)
  • מבצעים crdb-cli crdb update משמעותי

הקונפיג השולט: replication-backlog-size — כמה היסטוריה ה-master שומר בזיכרון לפעולת partial. ערך גדול יותר = פחות full syncs, אבל יותר זיכרון.

26. מהם 2 ה-Synchronization Modes של Active-Active?

לפי Configure distributed synchronization — docs.redis.io, ל-AA databases יש 2 synchronization modes:

  1. Centralized (ברירת מחדל) — כל database מקבל proxy endpoint יחיד שמטפל הן בתעבורת קליינט והן בתעבורת ה-sync בין הקלאסטרים.
  2. Distributed — כל proxy endpoint זמין יכול לטפל בתעבורת sync. משפר throughput ומוריד latency של sync traffic, קריטי ל-AA databases עם high throughput.

תנאי מקדים ל-Distributed: ה-proxy policy של ה-BDB חייב להיות all-nodes או all-master-shards (לא single) — כי ב-single יש רק proxy endpoint אחד מלכתחילה.

איך מגדירים: דרך REST API, rladmin, או UI ברמת ה-BDB. הגדרה ברמת database, לא cluster.

אל תבלבל עם sync types (partial vs full): המודים של synchronization mode מדברים על איך תעבורת ה-sync מנותבת בין endpoints; sync types מדברים על איך הנתונים מועברים בכל סשן (raw operations vs RDB).

טיפ לבחינה: 26 שאלות מקיפות. כדאי לחזור על הטאבים BDB Architecture, שירותים, ו-פורטים פעמיים. הנושאים החשובים: 3 proxy policies, 2 placement policies, Watchdog (Node + Cluster) ולא Sentinel, BDB כמושג, partial vs full sync, ופורטים מדויקים מהתיעוד הרשמי.
